## 二、Git协作规范

### 2.1 分支策略

| 分支 | 用途 | 保护状态 |
|------|------|----------|
| `main` | 稳定版本，最终交付代码 | ✅ 受保护（不能直接推送、不能删除） |
| `develop` | 开发主分支，日常协作 | ❌ 未保护（可自由推送） |

### 2.2 工作流程

#### 1. 每天开始工作前，拉取最新代码
> git checkout develop
git pull origin develop

#### 2. 创建功能分支（可选，小改动可直接在develop开发）
> git checkout -b feature/功能名称

#### 3. 开发完成后，提交代码
> git add .
git commit -m "类型(范围): 简要描述"

#### 4. 推送到远程
> git push origin develop

### 2.3 提交信息规范
| 类型 | 说明 | 示例 |
|------|------|------|
| `docs` | 文档相关 | docs (req): 更新需求分析报告 |
| `feat` | 新功能 | feat (backend): 添加用户认证 API |
| `fix` | Bug 修复 | fix (frontend): 修复登录页面报错 |
| `chore` | 构建 / 工具 | chore: 更新.gitignore |
| `test` | 测试相关 | test: 添加漏洞上报单元测试 |

### 2.4 提交频率要求
```
每人每周至少3次增量提交
不要一次性提交全部代码
提交前确保代码可运行
```
## 三、开发规范

### 3.1 后端规范（Django）
```
# 代码风格：遵循PEP 8
# 使用black格式化代码：black .
# 使用flake8检查代码：flake8 .
```

### 3.2 前端规范（React/Next.js）
```
# 代码风格：使用ESLint + Prettier
# 组件命名：PascalCase（如：VulnerabilityList.jsx）
# 文件命名：kebab-case（如：vulnerability-service.js）
```

### 3.3 测试规范
核心功能必须有单元测试
提交前运行测试：python manage.py test
测试覆盖率目标：≥70%

## 四、协作注意事项

### 4.1 必须遵守
```
禁止直接推送到 main 分支（已设置保护）
每天开始工作前先 git pull
提交前先 git status 确认文件
写清楚提交信息，不要用 "update"、"fix" 等模糊描述
```

### 4.2 建议遵守
```
使用 Issue 管理任务
重大功能使用 Pull Request 审查
遇到冲突及时沟通解决
```
### 4.3 紧急情况处理
如果误推送到 main，联系组长处理
如果出现合并冲突：
```
git checkout develop
git pull origin develop
git merge feature/xxx
```
手动解决冲突后：
```
git add .
git commit -m "merge: 解决合并冲突"
git push origin develop
```
## 五、环境配置

### 5.1 后端环境

创建虚拟环境
```
python -m venv venv
source venv/bin/activate  # Linux/Mac
venv\Scripts\activate     # Windows
```
安装依赖
```
pip install -r backend/requirements.txt
```
### 5.2 前端环境
安装依赖
```
cd frontend
npm install
```
启动开发服务器
```
npm run dev
```
